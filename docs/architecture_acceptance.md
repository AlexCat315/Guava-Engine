# Guava Engine 架构与验收标准（对标 UE5/Blender）

本文是 Guava Engine 的架构与验收标准草案，目标对标 UE5 与 Blender 的能力和质量，面向 PC 平台，兼顾游戏与动画。本文强调“可验证的质量”，因此所有条目都必须可被测试、对照或回归验证。本文不是时间规划，不讨论项目进度，仅定义“必须达到”的标准与落地形态。

为避免“玩具化”与“局部胜利”，本文将范围、数据模型、渲染、动画、编辑器、工具链、性能与质量保证作为同等重要的系统。单点突破没有意义，必须能贯穿“导入 → 编辑 → 预览 → 运行 → 输出”的完整生产链条。

## 1. 目标与定位

### 1.1 终极目标
- 一个可以独立完成游戏与动画项目的引擎与编辑器体系。
- 编辑器与运行时完全一致，任何功能不允许仅在编辑器存在。
- 画面与动画质量达到行业级标准，允许与高质量参考对照。
- 具备可持续维护的工程架构与可扩展的插件体系。

### 1.2 非目标与边界
- 不做单次 Demo 或临时项目特例。
- 不做纯研究性质的“不可用功能”。
- 不依赖外部商业工具完成核心链路。
- 不允许核心系统在不同平台出现功能断层。

## 2. 质量基线与验证方法

### 2.1 基线定义
基线不是“能运行”，而是“可交付”。所有系统必须满足以下基础条件：
- 稳定性：不允许由正常用户流程触发崩溃。
- 可复现：相同输入应生成相同输出。
- 可诊断：任何错误必须具备可定位的日志和上下文。

### 2.2 验证方法
所有验收必须依托以下方式之一：
- 自动化测试（单元、集成、系统）。
- 黄金结果对比（例如渲染图像差分）。
- 性能指标与统计数据记录。
- 结构一致性校验（序列化比对、资产依赖分析）。

## 3. 架构原则

1. 单一数据真相  
   资产与场景数据只有一个来源，任何缓存都可被回滚或重建。
2. 运行时与编辑器一致  
   同场景、同输入、同渲染结果。
3. 可重复与可复现  
   资源导入与构建是确定性的，任何机器均可复现。
4. RenderGraph 驱动  
   以 Pass 明确构建渲染依赖，避免隐式状态。
5. 系统可扩展  
   插件必须能访问核心系统，不需要修改主干即可扩展。
6. 工具链内建  
   构建、诊断、验证、性能分析与崩溃记录是系统一部分。

## 4. 系统总览与模块边界

系统必须明确分为以下核心模块：
- 平台与核心（Application、Window、Input、JobSystem）。
- 资产系统（AssetRegistry、Import、Cook、Cache）。
- 场景与 ECS（Entity、Component、World、Serialization）。
- 渲染与 GPU（RenderGraph、Material、Lighting、PostFX）。
- 动画系统（Skeleton、Clip、Blend、Sequencer）。
- 物理与仿真（Collision、RigidBody、Constraints）。
- 脚本与运行时逻辑（Script、Events、Gameplay）。
- 编辑器（Dock、Inspector、Timeline、Material/Anim Editor）。
- 工具链（CLI、Validator、Profiler、Crash Dump）。

任何新增功能必须归属其中某一模块并遵循边界，不允许跨模块耦合。

## 5. 统一数据模型与版本管理

### 5.1 资产 ID 与注册
必须建立稳定的全局资产 ID（建议 128-bit UUID 或 Hash+Namespace 组合），并在资产注册表中维护。  
验收标准：
- 资产 ID 与文件路径解耦，支持重命名与移动。
- 资产 ID 在项目生命周期内不可改变。
- 删除资产必须记录引用关系并触发清理提示。

### 5.2 数据结构（建议格式）
需要定义以下最低结构，并允许扩展：
- AssetRecord: id, type, source_path, source_hash, import_settings_hash, dependency_ids, outputs, metadata, version.
- SceneRecord: scene_id, entity_list, component_payloads, asset_refs, version.
- AnimationClip: clip_id, skeleton_id, sample_rate, keyframes, curve_data, events, compression, version.
- MaterialDef: material_id, shader_graph, parameters, texture_slots, version.

### 5.3 版本与迁移
验收标准：
- 所有数据结构具备 schema 版本。
- 老版本资产可自动迁移或明确阻止加载并报告。
- 任何版本变化必须有迁移器与回滚策略。

### 5.4 序列化一致性
验收标准：
- 同一场景重复保存，序列化文件字节级一致。
- 同一资产在不同机器导入，生成的构建输出一致。
- 任何资产引用都必须可解析到唯一 ID。

## 6. 资产系统与管线

### 6.1 资产管线流程
导入流程必须是标准化管线：
1. Source Discovery：扫描原始源文件。
2. Import Stage：转换成内部中间格式。
3. Cook Stage：根据目标平台生成可运行资源。
4. Cache Stage：生成可复用缓存并记录依赖。

### 6.2 热重载
验收标准：
- 文件修改后，资源可在 2 秒内重新加载并生效。
- 热重载必须能正确刷新依赖与资源引用。
- 热重载必须记录失败日志并可回滚。

### 6.3 依赖追踪
验收标准：
- 任何资源可追溯到完整依赖链。
- 删除资源必须列出所有直接与间接引用。
- 依赖变化必须触发重新构建。

### 6.4 资产验证
验收标准：
- 每种资产有独立验证规则。
- 验证失败必须阻止进入生产库并提供详细原因。

## 7. 场景与 ECS

### 7.1 实体与组件设计
必须支持：
- 稳定实体 ID。
- 组件版本管理。
- 编辑器与运行时组件分离。

验收补充：

- 运行时仿真状态不允许常驻 `EditorState` 这类编辑器对象。
- 粒子、动画、物理等会随场景播放演化的数据，必须归属于运行时世界或等价运行时容器。
- 编辑器只允许持有选择、检查器缓存、工具状态和临时 UI 缓冲，不得成为运行时真相来源。

### 7.2 序列化与增量更新
验收标准：
- 场景可增量保存（避免整场景重写）。
- 增量保存必须保证一致性与回滚能力。

### 7.3 运行时性能
验收标准：
- ECS 迭代效率在 10 万实体时依然稳定。
- 支持批量组件更新与系统调度。

## 8. 渲染架构与 RenderGraph

### 8.1 RenderGraph 要求
RenderGraph 必须具备：
- Pass 节点定义、资源节点定义。
- 资源生命周期自动管理。
- Barrier 与同步自动推导。
- Pass 可视化输出（DOT/JSON）。

验收标准：
- 每个 Pass 具备明确输入/输出。
- Graph 可导出并可视化。
- Pass 统计数据可访问（CPU/GPU 时间）。

### 8.2 渲染流程建议（可扩展）
基础流程应包含：
- Depth Prepass
- Shadow Pass（Directional/Point/Spot）
- Base Pass（PBR）
- Lighting / Forward+ or Deferred
- PostFX（HDR、Bloom、Tonemap、Color Grade）
- UI Pass

### 8.3 GPU 资源管理
验收标准：
- GPU 资源创建与销毁可追踪。
- 支持资源池化与复用。
- 资源泄漏必须可检测。

## 9. 材质系统与 Shader

### 9.1 材质系统
必须支持：
- PBR 标准工作流（金属度/粗糙度）。
- 材质实例化。
- 纹理槽（BaseColor、Normal、AO、Metallic、Roughness、Emissive）。
- 透明与不透明管线。

### 9.2 Shader 管线
必须支持：
- Shader Graph 或统一描述格式。
- 变体编译与缓存。
- 自动生成绑定布局。

验收标准：
- 材质变化不需要手动改代码。
- Shader 编译失败有清晰日志。
- 变体缓存可复用。

## 10. 光照、GI 与阴影

### 10.1 光照模型
必须支持：
- Directional、Point、Spot。
- 多光源混合。
- IBL（环境光照）。

### 10.2 GI 与反射
目标能力：
- 至少实现屏幕空间 GI 或基于探针的间接光。
- 可扩展到更高质量 GI（如 Lumen 级别方案）。

### 10.3 阴影系统
必须支持：
- 级联阴影（Directional）。
- 软阴影或滤波策略。
- 阴影调试可视化。

验收标准：
- 阴影可控、无明显破碎与闪烁。
- 重要光源阴影必须稳定。

## 11. HDR、色彩与后处理

### 11.1 色彩空间
必须规范：
- 纹理输入区分 sRGB / Linear。
- 内部全线性渲染。
- 输出支持 ACES 或等价 Tonemap。

### 11.2 后处理链
必须具备：
- Bloom
- Exposure
- Color Grading
- LUT
- Gamma 输出

可扩展：
- TAA
- SSAO
- SSR
- DOF
- Motion Blur

验收标准：
- 后期链顺序明确、可配置。
- 所有后期效果可独立开关并稳定运行。

## 12. 动画系统

### 12.1 动画数据
必须支持：
- Skeleton + Clip
- 关键帧曲线
- 动画压缩（可选）

### 12.2 播放与混合
必须支持：
- Blend Tree
- 状态机
- 层级混合（Upper/Lower Body）

验收标准：
- 同一 clip 在编辑器与运行时一致。
- 混合无抖动、无跳帧。

## 13. 时间轴与 Sequencer

### 13.1 时间轴功能
必须支持：
- 轨道（Transform、Camera、Event、Animation）
- Clip 排列与混合
- 关键帧编辑与曲线调整

### 13.2 验收标准
- 时间轴驱动的输出可保存与回放。
- 镜头与事件系统可稳定执行。

## 14. 物理与仿真

### 14.1 物理系统
必须支持：
- 碰撞形状（盒、球、胶囊、网格）
- 刚体与约束
- 物理调试可视化

验收标准：
- 同一输入结果一致。
- 编辑器与运行时物理一致。

## 15. 脚本与运行时逻辑

### 15.1 脚本系统
必须支持：
- 热重载
- Debug 工具
- 事件系统

验收标准：
- 脚本修改后可即时生效。
- 脚本与动画/物理/渲染一致协作。

## 16. 编辑器架构与 UX

### 16.1 编辑器架构
必须支持：
- Dock 化布局与状态持久化
- 模块化面板
- Undo/Redo 全覆盖

验收补充：

- Undo/Redo 的主路径必须是命令化历史，而不是对所有编辑操作无差别整场景快照。
- 高频编辑路径，特别是 gizmo 拖拽、层级调整、创建删除与复制，不允许在每次提交时做整场景序列化热路径。
- 若存在快照回退，只能作为未细化入口的临时兜底，而不是长期主实现。

### 16.2 关键工具
必须拥有：
- Scene Hierarchy
- Inspector
- Asset Browser
- Material Editor
- Animation Editor
- Timeline / Sequencer

验收标准：
- 编辑器无崩溃、无数据丢失。
- 编辑器操作不影响运行时数据结构的一致性。
- 视口覆盖层、工具栏和 3D 区域之间必须有明确输入所有权，UI 拖拽不得穿透到相机或场景操作。
- 平移、旋转、缩放工具必须基于单次手势起点的累计偏移计算，吸附只允许作用于输出值，不得逐帧污染内部状态。
- 鼠标滚轮、相机观察、物体选择和 gizmo 拖拽必须能稳定共存，不能出现因为 UI 焦点判断波动导致的随机失效。
- 编辑器交互手感参数必须可集中配置，不允许长期散落为多个无来源的魔法数字。

## 23. 具体验收场景与基准用例

为避免“功能存在但不可用”，必须建立固定的基准场景与验收用例。每个场景必须作为回归测试的一部分。

### 23.1 渲染基准场景
场景要求：
- 室内场景：包含多光源、反射材质、透明材质、阴影。
- 室外场景：包含大范围地形、植被、天空盒、动态阴影。
- 材质压力场景：至少 200 种材质实例，包含不同粗糙度与金属度。
验收标准：
- 渲染输出与参考图像差分在阈值内。
- Pass 时间统计稳定，避免异常抖动。

### 23.2 动画基准场景
场景要求：
- 单角色骨骼动画播放。
- 多角色混合（至少 20 个角色）。
- 时间轴驱动的镜头与事件。
验收标准：
- 动画与编辑器预览一致。
- 性能与帧率在可控范围。

### 23.3 物理基准场景
场景要求：
- 1000 个刚体堆叠。
- 复杂碰撞网格场景。
验收标准：
- 物理结果可复现。
- 不出现穿透、爆炸式抖动。

### 23.4 工具链基准场景
场景要求：
- 完整导入 → 编辑 → 保存 → 运行 → 导出流程。
验收标准：
- 资产依赖无丢失。
- 输出包可独立运行。

## 24. 资产格式与存储约定（建议标准）

为了保证可维护性，建议统一资源目录结构：
- assets/source/ 原始源文件（可版本控制）。
- assets/derived/ 中间格式（可重建）。
- assets/cooked/ 运行时资源（平台相关）。
- assets/cache/ 导入与构建缓存。

### 24.1 资产元数据文件
每个源资产必须有对应元数据文件（例如 .meta）：
内容至少包括：
- AssetId
- Importer 类型
- 版本号
- 依赖列表
- 构建参数哈希

### 24.2 构建缓存策略
建议缓存 Key 组成：
- SourceHash
- ImportSettingsHash
- ToolVersion
- PlatformTag
验收标准：
- 相同 Key 必须命中相同缓存结果。
- Cache 失效必须准确触发重建。

## 25. 渲染验证与图像一致性

### 25.1 图像对比标准
必须建立黄金图像库：
- 每个基准场景至少保留一组参考图像。
- 采用像素差分，允许低于 1% 噪声阈值。

### 25.2 渲染输出一致性
验收标准：
- 同一场景在编辑器/运行时输出一致。
- 不允许出现材质或光照缺失。

### 25.3 渲染异常检测
必须可检测：
- NaN 或 Inf 像素输出。
- 超出 HDR 范围的异常峰值。
- 资源绑定缺失。

## 26. 编辑器与运行时一致性协议

### 26.1 一致性原则
- 编辑器仅提供 UI，不改变渲染与逻辑。
- 编辑器与运行时使用同一渲染管线与资源系统。

### 26.2 验收标准
- 同一场景用编辑器预览与运行时输出一致。
- 编辑器 UI 关闭后，运行时结果不变。
- 编辑器专用数据不能进入运行时构建包。

## 27. 工程级稳定性与安全

### 27.1 崩溃处理
验收标准：
- 崩溃时必须生成完整日志。
- 日志包含时间戳、模块、调用栈。

### 27.2 内存安全
验收标准：
- 无明显内存泄漏。
- 大型场景可在限定内存内运行。

### 27.3 数据完整性
验收标准：
- 不允许出现无效引用或“悬空资源”。
- 每次保存后可通过一致性校验。
