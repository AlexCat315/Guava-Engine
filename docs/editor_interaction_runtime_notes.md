# Guava Engine 编辑器交互、历史与运行时数据说明

本文记录当前仓库中已经落地的编辑器稳定性与数据所有权调整，重点覆盖 Undo/Redo、视口交互、变换工具与 VFX 运行时。它不是路线图，而是“当前实现为什么这么做、已经做到哪里、还缺什么”的实现说明。

## 1. 适用范围

本文只描述以下几类问题：

- 高频编辑操作的历史记录形态
- 编辑器状态与运行时仿真状态的边界
- 视口输入隔离与 UI 事件穿透防御
- 平移、旋转、缩放工具的累计偏移与吸附规则
- 相关可配置项与已知剩余问题

## 2. 当前实现摘要

当前代码已经完成以下收敛：

- Undo/Redo 已切换到命令栈驱动，主通道不再依赖“每次操作都整场景快照”
- VFX 运行时已从 `EditorState` 剥离，归属 `World`
- 粒子运行时存储已切换到 `std.MultiArrayList`，以 SoA 布局承载高频更新
- 视口交互已区分 UI 覆盖层与 3D 区域，避免大部分输入穿透
- 变换吸附改为“基于单次拖拽起点的累计偏移”，不再逐帧污染实际变换
- 常用交互手感参数已提升为 `EditorState` 可配置字段，而不是散落在函数里的裸常量

## 3. Undo/Redo：命令栈优先，快照回退兜底

### 3.1 目标

高频编辑操作不能继续依赖整场景序列化，否则在场景规模增大后会把内存和交互延迟一起拖垮。当前实现的目标是：

- 常见实体编辑走增量命令
- 未细化的低频路径允许暂时走快照回退
- 对外仍保持统一的 Undo/Redo 入口

### 3.2 当前命令模型

当前 `EditorCommand` 是一个联合类型，包含两类命令：

- `subtree_delta`
- `scene_snapshot`

其中：

- `subtree_delta` 用于实体创建、删除、复制、重父子、重命名、gizmo 变换等“可局部表达”的操作
- `scene_snapshot` 是回退兜底，只给尚未细化到增量命令的入口使用

这意味着当前历史系统已经不是“全量快照历史”，而是“命令历史 + 快照兜底”。

### 3.3 已覆盖的高频路径

当前已经适合走增量命令的路径包括：

- 实体创建
- 实体删除
- 实体复制
- 层级重排与重父子
- 变换工具提交

这样做的直接收益是：

- 高频拖拽不再把整场景序列化当热路径
- 命令语义更接近实际编辑动作，后续扩展更容易
- Undo/Redo 与选择集恢复可以一起处理，而不是靠外部状态猜测

### 3.4 仍然保留的回退路径

当前仍有一部分低频编辑入口使用 `history.captureSnapshot(...)`。这不是旧系统残留到不可控状态，而是有意识保留的兜底策略：

- 先保证所有入口都能回退
- 再逐步把快照入口细化成增量命令

当前文档上的定义是：

- 快照回退是允许的临时手段
- 高频操作长期停留在快照回退上则不符合目标架构

## 4. VFX 运行时：编辑器只选实体，世界持有模拟状态

### 4.1 边界调整

VFX 的运行时状态已经不再归属于 `EditorState`。当前边界是：

- `EditorState` 只关心编辑器选择、UI 状态与工具状态
- `World` 持有 `vfx_runtime_emitters`
- VFX 运行时生命周期跟随场景和实体，而不是跟随编辑器窗口状态

这样做的原因很直接：

- 编辑器不应该拥有运行时仿真真相
- 新建场景、加载场景、删除实体时，VFX 运行时应该跟随场景数据一起收敛
- 后续无论是播放模式还是纯运行时构建，都不应该依赖编辑器层对象

### 4.2 粒子存储

当前粒子运行时使用 `std.MultiArrayList(VfxRuntimeParticle)`，即：

- 定义上仍保持结构体可读性
- 内存上采用 SoA 布局
- 高频更新能直接读取 `position`、`velocity` 等连续字段切片

这为后续两类工作打了基础：

- 大规模粒子更新的缓存友好访问
- 未来的批量向量化或 SIMD 优化

### 4.3 生命周期要求

当前实现要求以下事件都要正确处理 VFX 运行时：

- 场景清空
- 场景加载
- 实体销毁
- emitter 剪枝

约束很明确：

- 不允许删除实体后留下孤儿 emitter
- 不允许切场景后沿用旧 runtime 粒子
- 不允许编辑器关闭某个面板就影响 runtime 所有权

## 5. 视口交互：输入隔离优先于“看起来能点”

### 5.1 目标

视口交互需要满足两个条件：

- UI 覆盖层能可靠拦截自己的输入
- 3D 视口的选择、相机和 gizmo 不会误吃 UI 操作

换句话说，核心不是“按钮能不能点”，而是“输入所有权有没有被正确隔离”。

### 5.2 当前收敛点

当前已落地的行为包括：

- 鼠标输入会进入编辑器自己的 `InputState`
- 键盘是否继续下传仍会参考 ImGui capture 状态
- 视口覆盖层会维护 `viewport_overlay_hovered`
- 如果拖拽明确起始于 UI，会设置 `manipulation_started_from_ui`，阻止这次拖拽穿透到 3D 操作
- FPS 展示支持三态：`Viewport`、`Status Bar`、`None`
- 视口调试层默认关闭，需要时可手动开启

### 5.3 当前防线的边界

当前实现已经比“单纯用 `isItemHovered()` 判定一帧”更稳，但它仍不是最终形态。现阶段属于：

- 已经有 UI 起点锁存
- 仍然部分依赖 ImGui hover 信息

这意味着：

- 常见点击和拖拽穿透已经能被挡住
- 但若后续要把覆盖层做得更复杂，仍应继续升级到更明确的 active item / window capture 机制

## 6. 变换工具：累计偏移是硬约束，不是优化项

### 6.1 为什么必须改

如果每一帧都拿“上一帧已经被吸附后的结果”当下一帧起点，那么：

- 小幅慢拖会被 snap 一帧一帧抹掉
- 用户必须在单帧内甩出足够大的鼠标增量才看得到结果
- 平移、旋转、缩放三类工具都会表现出明显的卡阈值感

这个问题不是手感问题，而是数学模型错误。

### 6.2 当前规则

当前变换工具遵循以下规则：

1. 在拖拽开始时记录 `manipulation_origin`
2. 单次操作过程中累计鼠标偏移
3. 每一帧都从 `manipulation_origin` 重新计算目标变换
4. Snap 只作用于“这次计算出的绝对目标值”
5. 绝不把被 snap 的结果当成下一帧原点

这套规则同时适用于：

- `applyTranslate`
- `applyRotate`
- `applyScale`

### 6.3 当前实现上的关键状态

当前编辑器状态中已经存在以下字段用于支撑这套逻辑：

- `manipulation_origin`
- `manipulation_drag_active`
- `manipulation_drag_accumulator`
- `manipulation_accumulated_delta`

核心约束是：

- 原点只在本次拖拽开始时重置
- 拖拽结束或取消时清理累计状态
- 变换结果必须能从“原点 + 累计输入”完整重建

### 6.4 吸附的正确含义

吸附不应该修改拖拽累计量，它只应该修改最终输出值：

- 平移：围绕起始位置按步长量化
- 旋转：围绕起始欧拉角按步长量化
- 缩放：围绕起始缩放按步长量化，再做合法范围钳制

因此，吸附在这里是“输出整形”，不是“状态推进”。

## 7. 参数化：先脱离魔法数字，再谈外部配置

当前已从函数体中抽出的关键交互参数包括：

- `camera_boost_multiplier`
- `translation_drag_sensitivity`
- `rotation_drag_sensitivity`
- `scale_drag_sensitivity`

这一步的意义是：

- 至少先把交互手感从实现细节里剥出来
- 测试和调参不再需要改算法本体
- 后续迁移到 JSON / 用户设置时，字段已经有稳定落点

但要明确一点：当前还只是“状态内参数化”，还没有完成“外部数据驱动配置”。

## 8. 已知剩余问题

当前仍有以下问题没有完全收口：

- 视口覆盖层事件防御仍有一部分依赖 `isItemHovered()`，需要继续抽成统一的 capture 规则
- 交互参数虽然已提升为字段，但还没有统一迁移到外部配置文件
- 一些 UI 颜色和样式值仍然散落，尚未集中成主题或常量表
- 低频编辑入口仍存在 `scene_snapshot` 回退路径，尚未全部细化为增量命令

## 9. 后续建议顺序

建议下一轮继续按下面顺序推进：

1. 统一视口 overlay capture 规则，减少对 hover 瞬时状态的依赖
2. 把交互参数迁移到编辑器配置对象，并支持加载与保存
3. 继续把 inspector / asset / material 等低频入口改为显式命令
4. 将 UI 颜色、按钮状态与 overlay 样式集中到主题层

## 10. 文档结论

当前编辑器相关代码已经完成一轮关键收敛：

- 历史系统不再只有全量快照
- VFX 运行时已经回归运行时世界
- 视口输入比之前更可控
- 变换吸附数学模型已经从“逐帧污染”改成“基于起点的累计计算”

这套基础已经足够支撑下一步继续做：

- 更细颗粒的命令系统
- 更稳健的 overlay 输入隔离
- 真正的数据驱动编辑器配置
